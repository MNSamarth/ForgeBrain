# ForgeBrain Backend Architecture Review (Maintainability)

Scope: `backend/` (Spring Boot codebase) reviewed for long-term maintainability, not feature completeness.

## Findings

| Issue | Severity | Why it matters | Recommended improvement |
| --- | --- | --- | --- |
| `PipelineOrchestratorImpl` uses a hardcoded `if/else` stage chain and explicit nullable checks per artifact | High | Adding remaining stages (Voice, Subtitle, Assets, Renderer, Reviewer, Publishing) will grow a single orchestrator decision block and increase regression risk when stage order or transitions evolve | Move to a stage-handler registry/state machine (`Map<PipelineStage, StageHandler>`), where each handler declares `canRun` + `run`, and orchestration only manages transitions |
| `PipelineContext` is a mutable "all-artifacts" bag with many nullable fields | High | Temporal coupling grows with pipeline size; invalid state combinations become easier to create and harder to test as code grows to 50k+ LOC | Replace with stage-scoped immutable state objects (or sealed hierarchy) so legal transitions are encoded in types, not null checks |
| `MemoryServiceImpl` combines storage adapter concerns (file I/O, serialization, path handling, caching) with domain mutation logic (`markTopicInProgress`, content-fragment tracking) | High | Violates SRP; storage migration (Firestore) and domain rule changes will collide in one class | Split into `MemoryRepository` (persist/load raw state) and `MemoryDomainService` (business transitions); keep orchestration depending on domain service |
| `TopicSelectorImpl` has multiple responsibilities: candidate gating, scoring, curriculum flattening/indexing, reasoning text generation, and embedded weight profiles | High | 300+ line policy engine is difficult to reason about, change safely, and tune; policy changes will require touching unrelated code paths | Extract: `CandidateGate`, `TopicScorer`, `SelectionExplainer`, and `ModeWeightsProvider` (externalized weights source) |
| Score weights are duplicated in code (`ModeWeights.forMode`) while docs reference `brain/topic-ranking.json` as the source of truth | Medium | Drift risk: behavior can silently diverge from spec/config as ranking evolves | Load weights from versioned config (or generate constants from one source) and add startup validation/version check |
| `VertexAiResearchServiceImpl` and `VertexAiLessonServiceImpl` instantiate fallbacks directly (`new ResearchServiceImpl(...)`, `new LessonServiceImpl()`) | High | Bypasses DI lifecycle and makes cross-cutting behavior (instrumentation, policy wrappers, testing overrides) harder; encourages hidden dependency wiring | Use constructor-injected fallback strategy interfaces (`ResearchService fallback`, `LessonService fallback`) with explicit qualifier names |
| Vertex + heuristic services duplicate decision/helper logic (`deriveTeachingStyle`, confidence-note composition pattern, local text helpers such as `lowerFirst`, rounding helpers) | Medium | Duplication increases inconsistency risk when behavior changes and creates maintenance overhead | Extract shared helpers/policies (e.g., `TeachingStylePolicy`, `TextFormattingUtils`, `ConfidenceNoteFactory`) and reuse them across implementations |
| `PipelineCommandLineRunner` catches broad `Exception` and converts failures into logs only | Medium | Startup-run failures become easier to miss in automated environments; weak failure semantics become harder to manage at scale | Catch expected exception families explicitly, publish structured failure result/metric, and make fail-fast behavior configurable |
| Package layout is mostly technical-layer based (`services`, `models`, `pipeline`) while stage-specific logic is growing | Medium | As lines scale, finding all code for one stage requires cross-package hunting, slowing onboarding and refactoring | Organize by bounded context/stage first (e.g., `pipeline.research`, `pipeline.lesson`, `pipeline.script`) with shared infra in `infrastructure.*` |
| `services` package mixes stage orchestration logic with prompt-building utilities (`LessonPromptBuilder`, `ResearchPromptBuilder`) | Medium | Blurs boundaries between application services and LLM prompt infrastructure | Move prompt builders into a dedicated package (e.g., `infrastructure.vertex.prompt` or `pipeline.<stage>.prompt`) |
| Persistence direction is split between current file-based state (`MemoryServiceImpl`) and placeholder entity/repository model (`entities`, `repositories`) without an explicit adapter boundary | Medium | Migration to Firestore can produce duplicated pathways and temporary coupling unless a single persistence abstraction is established now | Define one storage port for memory/pipeline run persistence and keep file + Firestore implementations behind that port |
| Naming currently communicates implementation history more than role (`VertexAi*ServiceImpl` as primary services, `ResearchServiceImpl`/`LessonServiceImpl` as fallback-only heuristic services) | Low | Naming ambiguity slows comprehension and increases accidental misuse as team size/code size grows | Rename for role clarity: e.g., `VertexResearchService`, `HeuristicResearchService`, `VertexLessonService`, `HeuristicLessonService`, optionally with a `Fallback*` decorator |

## Notes

- This review intentionally excludes unimplemented pipeline stages and focuses on maintainability of existing backend code.
- No feature changes were made as part of this review.
